Overview

What is the Parquet & Polaris Reader?

The Parquet & Polaris Reader (also called Datalake Reader or GCS Parquet Explorer) is a web-based SQL interface for querying SAP data stored in cloud data lakes. It allows users to run SQL queries against Parquet files in Google Cloud Storage, Azure Blob Storage, and AWS S3 (via Polaris/Iceberg catalogs) — all without installing any local tools.

It is built on:

  • DuckDB — an in-process analytical SQL engine optimized for columnar data
  • PyArrow — Apache Arrow implementation for Python, used for Parquet file I/O
  • Python HTTPS server — the same web server that hosts the SAP Skills Portal
URL: https://sapidesecc8.fivetran-internal-sales.com/datalake_reader/ — Login with your Fivetran email + password.
Architecture

How It Works

The system has three layers:

LayerComponentPurpose
Web Servergcs_explorer_server.pyPython HTTPS server (port 443), handles authentication, serves UI, routes API calls
SQL EngineDuckDB subprocessExecutes SQL queries in a separate process, communicates via stdin/stdout JSON protocol
Data AccessDuckDB extensions (httpfs, iceberg, azure)Reads Parquet/Iceberg data directly from cloud storage over HTTPS
Stateless design: DuckDB runs in-memory with no persistent local database. Tables are loaded on-demand from cloud storage when queried. If the process restarts, tables are re-loaded transparently.

Request Flow

  1. User types SQL in the browser UI
  2. Browser sends POST to /datalake_reader/api/query
  3. Web server authenticates the session cookie
  4. Server calls run_sql(query) which sends the query to the DuckDB subprocess
  5. DuckDB executes the query — if a table is missing, it auto-loads from GCS
  6. Results (columns + rows) are returned as JSON to the browser
  7. Browser renders the results as a table with pagination

DuckDB Engine

What is DuckDB?

DuckDB is an in-process OLAP (Online Analytical Processing) database engine. Unlike traditional databases (PostgreSQL, MySQL), DuckDB:

  • Runs in-memory — no separate server process, no disk-based storage needed
  • Columnar storage — optimized for analytical queries (aggregations, scans, joins on large tables)
  • Reads Parquet natively — can query Parquet files directly without importing data
  • Supports remote files — reads from GCS, S3, Azure via the httpfs extension
  • Standard SQL — PostgreSQL-compatible SQL dialect

How DuckDB Runs in the Reader

DuckDB runs as a subprocess managed by the web server:

  1. At server startup, a DuckDBWorker subprocess is spawned
  2. The subprocess loads extensions: httpfs (GCS/S3), iceberg (Polaris catalogs), azure (Azure Blob)
  3. The web server sends SQL commands via JSON over stdin/stdout pipes
  4. The subprocess executes queries and returns column names + row data
  5. If a table doesn't exist, the server auto-loads it from cloud storage (auto_load_table)
Auto-loading: When you query SELECT * FROM MARA and the table doesn't exist in DuckDB, the server searches GCS for a matching Parquet file (e.g., gs://bucket/MARA/*.parquet), creates a view pointing to it, and re-runs the query. This is transparent to the user.

PyArrow

What is PyArrow?

PyArrow is the Python implementation of Apache Arrow, an in-memory columnar data format designed for high-performance analytics. It serves as the bridge between DuckDB and the Parquet file format.

  • Apache Arrow — a language-independent columnar memory format for flat and hierarchical data. It enables zero-copy data sharing between processes and languages (Python, Java, C++, R).
  • PyArrow — the Python bindings for Arrow. Provides fast Parquet read/write, IPC (inter-process communication), and integration with pandas, NumPy, and DuckDB.

How We Use PyArrow

In the Parquet & Polaris Reader, PyArrow serves several critical roles:

RoleDescription
Parquet I/OReads and writes Parquet files. When DuckDB uses read_parquet(), it relies on PyArrow's Parquet implementation under the hood for file format parsing, column decoding, and decompression.
Schema inferenceAutomatically detects column names and data types from Parquet file metadata without reading the full dataset. This enables the Reader to show table schemas instantly.
Data type mappingMaps Parquet data types (INT32, INT64, BYTE_ARRAY, TIMESTAMP, DECIMAL) to Python/DuckDB types. Handles SAP-specific types like packed decimals and date formats.
CompressionSupports all Parquet compression codecs: Snappy (default for Fivetran output), Gzip, Zstd, LZ4, and uncompressed. Decompression is done natively in C++ for maximum speed.
Predicate pushdownReads Parquet row group statistics (min/max values) to skip irrelevant data blocks. When DuckDB pushes a WHERE filter, PyArrow uses these statistics to avoid reading unnecessary row groups.
Column pruningOnly reads the columns referenced in the SQL query. A SELECT MATNR, MAKTX FROM MARA on a 50-column table only downloads 2 columns from cloud storage.
Activity loggingThe activity logger uses PyArrow to write individual Parquet files to gs://sap_cds_dbt/portal_log/ for the activity audit trail.

PyArrow vs DuckDB Native Parquet

DuckDB has its own built-in Parquet reader (parquet extension), but PyArrow provides additional capabilities:

FeatureDuckDB NativePyArrow

Parquet writeYesYes (more format options)

In practice: DuckDB uses its native Parquet reader for SQL queries (faster for analytical workloads), while PyArrow is used for metadata inspection, schema discovery, and writing activity log Parquet files.

Parquet Files

What is Parquet?

Apache Parquet is a columnar file format designed for efficient data storage and retrieval. Key properties:

  • Columnar — data is stored by column, not by row. This makes analytical queries (SELECT specific columns, aggregations) extremely fast because only the needed columns are read from disk/network.
  • Compressed — columns are compressed individually using Snappy, Gzip, or Zstd. Typical compression ratios are 5-10x vs CSV.
  • Schema-embedded — the file contains its own schema (column names, types), so no external schema definition is needed.
  • Partitioned — large datasets are split into multiple Parquet files (partitions), enabling parallel reads and predicate pushdown.

How the Reader Queries Parquet

When DuckDB queries a Parquet file from GCS:

  1. DuckDB's httpfs extension makes HTTPS range requests to GCS
  2. It reads only the Parquet metadata footer first (column names, row groups, statistics)
  3. Based on the query's WHERE clause, it applies predicate pushdown — skipping row groups that don't match
  4. Only the required columns are downloaded (column pruning)
  5. Data is decompressed and processed in-memory

This means querying SELECT COUNT(*) FROM large_table WHERE date > '2026-01-01' on a 10GB Parquet file might only download a few MB of data.

GCS Parquet Structure

SAP CDS View extraction writes Parquet files to GCS in this structure:

gs://<bucket>/<connector_id>/<schema>/<table_name>/
    part-00000-XXXXX.parquet
    part-00001-XXXXX.parquet
    ...

The reader creates a DuckDB view that reads all partitions:

CREATE VIEW "TABLE_NAME" AS
SELECT * FROM read_parquet('gs://bucket/path/TABLE_NAME/*.parquet');
Polaris Catalog Integration

What is Polaris?

Polaris is an open-source catalog service for Apache Iceberg tables. It provides a REST API that manages table metadata, enabling tools like DuckDB to discover and query Iceberg tables without knowing the underlying file paths.

Fivetran uses Polaris as the metadata catalog for SAP data delivered to cloud data lakes. When Fivetran extracts data from SAP, it writes Iceberg tables to S3/GCS/Azure and registers them in a Polaris catalog.

How the Polaris Integration Works

  1. OAuth authentication — The reader authenticates to the Polaris catalog using OAuth2 client credentials (stored in the encrypted vault on sapidesecc8 and in /usr/sap/gcs_explorer.env)
  2. Catalog discovery — The reader calls the Polaris REST API to list available namespaces and tables
  3. Iceberg metadata — For each table, Polaris returns the Iceberg metadata location (a JSON file pointing to the actual data files)
  4. Vended credentials — Polaris provides short-lived S3/GCS/Azure credentials (vended credentials) so DuckDB can access the data files directly
  5. Query rewriting — The reader's _rewrite_aws_query() function translates user SQL into DuckDB's iceberg_scan() function with the vended credentials
Vended credentials are temporary, scoped access tokens that Polaris generates on-the-fly. They grant read-only access to specific S3/GCS paths for a limited time. This means no long-lived cloud credentials are stored on the server — access is always proxied through Polaris.

Polaris Query Flow

User SQL:     SELECT * FROM aws.namespace.table_name LIMIT 10

Rewritten to: SELECT * FROM iceberg_scan(
                's3://bucket/path/metadata/v1.metadata.json',
                access_key_id = 'ASIA...',        -- vended credential
                secret_access_key = '...',         -- vended credential
                session_token = '...',             -- vended credential
                region = 'us-east-1'
              ) LIMIT 10

The user never sees the rewritten query — they just use standard SQL table names.

Data Sources

GCS (Google Cloud Storage)

Direct access to Parquet files in GCS buckets using application default credentials configured on sapidesecc8.

-- Browse a GCS bucket
SELECT * FROM read_parquet('gs://bucket-name/path/*.parquet');

-- Or use auto-loaded tables (just query by name)
SELECT * FROM MARA LIMIT 10;

AWS S3 (via Polaris/Iceberg)

Access Iceberg tables in S3 through the Polaris catalog. Credentials are vended automatically.

-- List tables in an AWS namespace
SELECT * FROM aws.my_namespace.my_table LIMIT 10;

Azure Blob Storage

Access Iceberg tables in Azure through the Polaris catalog, or direct Parquet reads via the azure extension.

-- Via Polaris
SELECT * FROM azure.my_namespace.my_table LIMIT 10;

-- Direct (requires credentials)
SELECT * FROM read_parquet('az://container/path/*.parquet');
Operations

Start / Stop / Restart

The Parquet Reader runs as part of the gcs-explorer.service systemd unit (the same process that serves the SAP Skills Portal).

# Check status
systemctl status gcs-explorer

# Restart (also restarts the portal)
systemctl restart gcs-explorer

# View logs
tail -50 /var/log/gcs_explorer.log

Or use the Start/Stop/Restart buttons on the cockpit page or the Monitoring page.

Note: Stopping the service also stops the SAP Skills Portal. They are the same process.

Upgrade

# Upgrade DuckDB + PyArrow
/root/miniconda/bin/pip install --upgrade duckdb pyarrow

# Restart to load new versions
systemctl restart gcs-explorer

# Verify
/root/miniconda/bin/python3 -c "import duckdb; print(duckdb.__version__)"

Or use the Upgrade DuckDB button on the cockpit page.

Troubleshooting

Common Issues

IssueFix
Query returns "Table not found"The table hasn't been auto-loaded yet. Browse the GCS bucket in the UI first, or use read_parquet('gs://...') directly.
Slow queryAdd LIMIT to avoid scanning entire datasets. Use WHERE clauses on partitioned columns. Check DuckDB trace log for query details.
Polaris table access deniedCheck OAuth credentials in the Polaris settings panel. Vended credentials may have expired — retry the query.
"httpfs extension not loaded"Restart the service: systemctl restart gcs-explorer. Extensions are loaded at startup.
Server not respondingCheck systemctl status gcs-explorer. View /var/log/gcs_explorer.log for errors. The DuckDB subprocess may have crashed — restart resolves this.
After DuckDB upgrade, queries failClear cached tables by restarting. If extension API changed, check DuckDB release notes for breaking changes.
